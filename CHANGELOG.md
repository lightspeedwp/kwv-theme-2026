## [Unreleased]

### Changed (Shop archive ‚Äî 15 products per page)
- **`inc/woocommerce.php`** ‚Äî added a `loop_shop_per_page` filter returning **15** (was WooCommerce's default 16). The archive's Product Collection uses `inherit: true`, so its count comes from the main query / this filter rather than the block's `perPage`; 15 fills 5 complete rows on the 3-wide grid instead of leaving a single orphan tile on the last row. **Verified** on the live local shop: the results count reads "Showing 1‚Äì15 of 24 results" with 15 product tiles.

### Added (Mini Cart ‚Äî light/dark header badge block styles)
- **`inc/woocommerce.php`** ‚Äî registered two block styles on `woocommerce/mini-cart` (init priority 20, after WooCommerce registers the block): **Light Header** (`is-style-mini-cart-light`) and **Dark Header** (`is-style-mini-cart-dark`). The quantity badge needs opposite colours in each of the site's two headers to stay legible; WooCommerce folds the block `className` onto the front-end `.wc-block-mini-cart` wrapper, so the `.is-style-*` rules resolve.
- **`assets/styles/woocommerce.css`** ‚Äî split the single `.wc-block-mini-cart__badge` rule into a bare default plus two variant rules. `is-style-mini-cart-light` (light header) ‚Üí contrast (dark) badge with base (light) count; `is-style-mini-cart-dark` (dark header) ‚Üí base (light) badge with contrast (dark) count. The bare default keeps the previous dark-look colours so any Mini Cart without a style (e.g. the transparent header) still reads against a dark backdrop.
- **`patterns/header-light.php`** / **`patterns/header-dark.php`** ‚Äî applied `is-style-mini-cart-light` and `is-style-mini-cart-dark` to each header's Mini Cart block respectively.

### Added (Shop filters ‚Äî collapsible sections)
- **`assets/js/shop-filters.js`** (new) + **`inc/woocommerce.php`** ‚Äî each WooCommerce Product Filter section in the shop sidebar (Brands, Price Range, Type, Pack Size, Special Offers) now folds up by default with a +/- disclosure on its heading. The script scopes to `.kwv-shop-filters` (the `className` the `kwv/woo-product-archive` pattern puts on the `woocommerce/product-filters` block), turns each section heading into a `role="button"` disclosure control (Enter/Space + click, `aria-expanded`), and toggles a `kwv-filter-collapsed` class. Because the filters are an Interactivity API region (`data-wp-router-region`) that re-renders on filter changes ‚Äî wiping injected DOM ‚Äî interaction is **delegated** off the document against the stable `data-block-name` structure (never an injected class), a **MutationObserver** re-applies enhancement after each re-render, and open/closed state is remembered per section (keyed by `data-wp-key`, else heading text) so a section the shopper opened stays open across renders. Progressive enhancement: the container only gets `kwv-filters-collapsible` (which the CSS gates the fold on) once JS runs, so with no JS every section stays open and fully accessible. Enqueued front-end only, deferred, gated to `is_shop()`/`is_product_taxonomy()`.
- **`assets/styles/woocommerce.css`** ‚Äî the fold presentation, keyed off the sections' stable `data-block-name^="woocommerce/product-filter-"` structure (the `product-filter-` prefix + a `:has(> .wp-block-heading)` guard match the five real sections but not the `product-filters` container, the nested list/slider blocks, or the headingless active-filters chips). A **neutral-700** 1px divider sits below every visible section (skipping `[hidden]`/`.wc-block-product-filter--hidden` empty ones). The fold is a CSS-grid row-collapse (`grid-template-rows: auto minmax(0, 1fr)` ‚Üí `minmax(0, 0fr)`, transitioned 0.25s) which keeps the section full-width while folded so the CSS price slider still measures on expand; the content row is `overflow:hidden; min-height:0` with no padding (so it reaches a true zero ‚Äî the heading‚Üîcontent gap is a heading `margin-bottom` applied only when open). The +/- glyph is a single `::after` masked with an inline SVG (a `currentColor`-tinted plus, swapped for a minus SVG on expand) ‚Äî a whole-icon swap with no scaling bars or transforms, so there's no drift or odd motion. A `brand-500` `:focus-visible` outline marks keyboard focus. The filter-name headings also get `text-transform: none` (theme.json's `elements.heading` uppercases all headings; cased filter names read better) and a slight `spacing-10` left padding, and each section's content (checkbox list / price slider) gets a matching `spacing-10` left indent so it lines up with the label; the price slider additionally gets `spacing-10` right padding so its min/max handles (which sit half-outside the track) no longer overflow the sidebar edges. Checkbox-list items are set to `font-size 200`, and top-aligned (`align-items: flex-start` on both the `__label` and its `__text-wrapper`, plus a 2px top pad on the text) so a label that wraps to two lines keeps the checkbox and count on the first line while a single-line label still reads centred against the checkbox. WooCommerce's own chrome on the filters wrapper is neutralised too: the `.wc-block-product-filters__overlay` grey `border-top` + top padding/margin (a mobile-bar separator that reads as a stray light line on desktop) is removed, and the `.wc-block-product-filters__overlay-content` flex `gap` (a hard-coded block-gap fallback that ignored the editor and doubled up with our section spacing) is zeroed so spacing comes solely from each section's `padding-block` + divider. In CSS not a JSON `css` field because it needs pseudo-element glyphs, `:focus-visible` and JS-class-driven state. **Verified** headless against the live local shop: all five sections collapse flush by default (content height 0), the heading toggle expands/collapses with correct `aria-expanded`, the divider computes to `#505050` (neutral-700), and ‚Äî the key risk ‚Äî applying a brand filter re-rendered the region while the enhancement re-applied and the open "Brands" section stayed open.

### Fixed (Product Collection grid ‚Äî 300px minimum card width)
- **`assets/styles/woocommerce.css`** ‚Äî Product Collection cards could shrink too small on mid-size viewports. WooCommerce's responsive (shrinkColumns) grid (`.wc-block-product-template__responsive`) floors each column at **150px** (`minmax(max(150px, calc(<n>% - gap)), 1fr)`). Raised that floor to **`min(300px, 100%)`** across the `columns-2`‚Ä¶`columns-6` variants, mirroring WooCommerce's own formula so each pattern's intended column cap is preserved (archive/search stay 4-up, sidebar-archive 3-up) while `auto-fill` drops the column count as the container narrows ‚Äî cards now reflow instead of shrinking below 300px. `min(300px, 100%)` prevents horizontal overflow on phones whose content column is itself narrower than 300px (falls back to one full-width card). Added a matching `min-width: min(300px, 100%)` + `flex-grow: 1` floor for the non-responsive flex layout (`.is-flex-container > li`, e.g. the single-product "You may also like" related-products grid). Column gap stays WooCommerce's default 1.25em; only the pre-existing `row-gap: 2em` override is unchanged. This selector set already overrides WooCommerce (the existing `row-gap` override on the same selector wins), so no specificity/load-order change was needed. **Not** visually verified ‚Äî the local site's browser session was locked and the change is a deterministic mirror of WooCommerce's grid formula.
